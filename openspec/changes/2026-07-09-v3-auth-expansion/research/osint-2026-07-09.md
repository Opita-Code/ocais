## 1. AWS Cognito CVEs (2023-2026)

| CVE | Date | CVSS | Affected | Description | Lessons for OCAIS |
|---|---|---|---|---|---|
| **CVE-2023-22893** | 2023-02 | 9.8 | Strapi + Cognito | Auth bypass via forged tokens | **Don't trust third-party integrations blindly.** Verify tokens with the IdP's public key, not the IdP's SDK if the SDK has known issues. |
| **CVE-2024-28056** | 2024-04 | 5.3 | AWS Amplify CLI | Auth bypass when Auth component removed | **Deleted components don't mean deleted identity.** Migration paths must invalidate or re-validate identity state. |
| **CVE-2024-45037** | 2024-08 | 7.5 | AWS CDK | `CognitoUserPoolAuthorizer` grants broader access than intended via auth scopes | **Auth scope chains are tricky.** OCAIS v3.0 should expose primitive-level controls (not high-level "scope" abstractions) so consumers can't accidentally over-grant. |
| **CVE-2024-52311** | 2025-10 | 6.5 | data.all (Cognito consumer) | Tokens not invalidated on logout | **Logout must invalidate server-side.** OCAIS v3.0 needs explicit `revokeToken` primitive with server-side revocation list. |

### Pattern (all 4 CVEs)

- **Token lifecycle bugs**: forge, scope, invalidate
- **Integration misuse**: third-party auth libraries make wrong assumptions about Cognito's contract
- **Migration hazards**: deletion doesn't equal revocation

### OCAIS v3.0 design implications

1. **Token verification uses standard primitives** (RFC 7519 JWT, RFC 7515 JWS) — not vendor-specific
2. **Auth model is explicit, not abstracted**: consumer must specify `aud`, `iss`, `exp`, `nbf` per token
3. **`revokeToken` is a first-class primitive** — not a "logout" side-effect
4. **DDB key for token revocation list** — `REVOKE#<jti>` with TTL = `exp`

## 2. OWASP ASVS 4.0 Controls (auth + session management)

OCAIS v3.0 must meet **ASVS Level 2** (minimum for SaaS) and **Level 3** (for high-value apps).

### V2 — Authentication (relevant subset)

| ID | Requirement | OCAIS v3.0 primitive |
|---|---|---|
| V2.1.1 | Verify users against stored credentials | `passwordVerify` |
| V2.1.5 | Verify users have a password confirmation | `passwordHash` + `passwordVerify` |
| V2.1.7 | Enforce password strength | consumer-implemented (OCAIS exposes params) |
| V2.5.1 | Define and document all password recovery paths | `magicLinkVerify` + `magicLinkRequest` |
| V2.5.5 | Use secure channels for password recovery | consumer (SES, SendGrid) |
| V2.7.1 | Verify out-of-band authentication codes | `magicLinkVerify` (32-byte token) |
| V2.9.1 | Re-authenticate for sensitive operations | `passwordVerify` or `magicLinkVerify` |

### V3 — Session Management

| ID | Requirement | OCAIS v3.0 primitive |
|---|---|---|
| V3.2.1 | Generate session IDs with sufficient entropy | `signJWT` (EdDSA / HS256) |
| V3.2.2 | Use sufficient session ID length | 32-byte minimum (256 bits) |
| V3.2.3 | Bind session to client (cookie attributes) | `cookieSign` (defaults: HttpOnly, Secure, SameSite=Lax) |
| V3.2.4 | Use `SameSite=Lax` or `Strict` | `cookieSign` default |
| V3.3.1 | Invalidate session on logout | `revokeToken` + DDB key |
| V3.3.2 | Allow user to log out from all sessions | `revokeAllTokensForUser` |
| V3.4.1 | Set `Secure` on cookies sent over TLS | `cookieSign` default |
| V3.4.2 | Set `HttpOnly` on session cookies | `cookieSign` default |
| V3.4.3 | Set `SameSite` on session cookies | `cookieSign` default |
| V3.5.1 | Verify JWT alg matches expected (prevent `none` confusion) | `verifyJWT` (rejects `none`) |
| V3.5.2 | Verify JWT signature with expected key | `verifyJWT` (uses `jwksPublish`) |
| V3.5.3 | Verify JWT `exp`, `nbf`, `iss`, `aud` | `verifyJWT` (enforced) |
| V3.5.4 | Reject JWT with `kid` not in JWKS | `verifyJWT` |

### V6 — Stored Cryptography

| ID | Requirement | OCAIS v3.0 primitive |
|---|---|---|
| V6.2.1 | Use authenticated encryption for sensitive data | `cookieSign` (HMAC + AES-256-GCM) |
| V6.2.4 | Use approved password hashing (Argon2id, bcrypt, scrypt, PBKDF2) | `passwordHash` (Argon2id only) |
| V6.3.1 | Use cryptographically secure random for secrets | `crypto.getRandomValues` (built-in) |
| V6.4.1 | Use managed key management (KMS, Vault) for key storage | consumer (OCAIS delegates) |

### V7 — Error Handling and Logging

| ID | Requirement | OCAIS v3.0 primitive |
|---|---|---|
| V7.1.1 | Log auth events with sufficient context | structured log: `requestId, source, allow, latencyMs, error` |
| V7.1.2 | Don't log secrets | OCAIS redacts `password`, `token`, `cookie` |

## 4. Primitives Decision (own vs library)

| Primitive | Own (zero-dep) | Library | Decision | Rationale |
|---|---|---|---|---|
| **Magic-link token** | `crypto.getRandomValues` (32 bytes) | none | **Own** | Trivial. NIST SP 800-90A. |
| **Magic-link HMAC** | `crypto.subtle.sign('HMAC', ...)` | none | **Own** | 10 lines. Standard SHA-256. |
| **Magic-link storage** | storage adapter (consumer) | none | **Own (interface)** | OCAIS doesn't ship DDB adapter. User implements. |
| **JWT signing (HS256)** | `crypto.subtle` | none | **Own** | Standard RFC 7515/7519. ~30 lines. |
| **JWT signing (EdDSA)** | `@noble/ed25519` (peer-dep, optional) | `jose` (heavy) | **noble** | noble is pure-JS, audited, no deps. jose is 50KB+ and brings full JOSE features we don't need. |
| **JWT verification** | `crypto.subtle` + manual checks | `jose` | **Own** | Critical control: must reject `alg: none` (V3.5.1). Library hiding this is a risk. |
| **Password hashing** | **NO** | `@node-rs/argon2` (peer-dep, optional) | **@node-rs/argon2** | Native bindings, 5-10x faster than pure JS. OWASP recommends Argon2id. We refuse to reimplement memory-hard hashing. |
| **Cookie sealing** | `crypto.subtle` (HMAC + AES-GCM) | none | **Own** | Standard authenticated encryption. ~40 lines. |
| **Rate limit** | `Map<key, counter>` (in-memory) | none | **Own (in-memory) + storage adapter** | In-memory for single Lambda; storage adapter for cross-instance. |

### Zero-dep vs peer-dep policy

- **Zero runtime deps** by default. OCAIS works without any install beyond itself.
- **Optional peer-deps** for primitives where reimplementing would be a security bug:
  - `@noble/ed25519` for EdDSA JWT (optional — HS256 works without it)
  - `@node-rs/argon2` for Argon2id (optional — but recommended; password hashing without it is a footgun)
- **No transitive deps**. Each peer-dep is a leaf, no deps-of-deps.
- **No dev-deps in runtime**. `package.json` has `peerDependencies` for runtime, `devDependencies` for build.

## 6. Open Questions (Phase 0)

- [ ] Is the DDB revocation list sufficient, or do we need Redis for low-latency `isRevoked?` checks?
- [ ] Should `cookieSign` support JWE (encrypted JWT) or only JWS (signed)? Phase 1 default: JWS only. JWE in v3.1 if needed.
- [ ] Should `magicLinkVerify` support multi-tenant scoping (token per workspace)?
- [ ] Should OCAIS ship a DDB adapter as `@opita/ocais/storage-ddb` (peer package)?

## 7. Decision Summary

- **MVP primitives** (Phase 1): magic-link, JWT, cookie, errors
- **Storage-agnostic**: consumer implements `AuthStorage` interface
- **Zero-dep core**, **optional peer-deps** (noble/ed25519, @node-rs/argon2)
- **ASVS Level 2** minimum, **Level 3** for high-value paths
- **AWS Cognito + Auth0 + Clerk post-mortems** inform all design decisions
- **Better-Auth is a benchmark, not a dependency** — we learn from its primitives API but implement our own
